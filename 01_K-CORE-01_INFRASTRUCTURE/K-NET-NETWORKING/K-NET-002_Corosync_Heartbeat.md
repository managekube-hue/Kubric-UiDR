# K-NET-002 — Corosync Cluster Heartbeat Configuration

> **Cluster Name:** kubric-cluster  
> **Nodes:** pve-kubric-01, pve-kubric-02, pve-kubric-03  
> **Transport:** Unicast (udpu) — no multicast required  
> **Heartbeat Network:** 10.0.200.0/24 (storage VLAN, low-latency)

---

## 1. Proxmox Cluster Creation

### 1.1 Initialize on Node 1

```bash
# On pve-kubric-01 (first node)
pvecm create kubric-cluster \
  --link0 10.0.200.21 \
  --votes 1

# Verify
pvecm status
# Expected:
#   Cluster name: kubric-cluster
#   Quorum provider: corosync_votequorum
#   Nodes: 1
#   Node ID: 1
```

### 1.2 Join Node 2

```bash
# On pve-kubric-02
pvecm add 10.0.100.21 \
  --link0 10.0.200.22 \
  --fingerprint $(ssh root@10.0.100.21 pvecm status | grep -v "^$")

# Enter root password for pve-kubric-01 when prompted
```

### 1.3 Join Node 3

```bash
# On pve-kubric-03
pvecm add 10.0.100.21 \
  --link0 10.0.200.23

# Enter root password for pve-kubric-01 when prompted
```

### 1.4 Verify Full Cluster

```bash

pvecm status
```

## 2. Corosync Configuration

### 2.1 /etc/pve/corosync.conf

This file is auto-generated by `pvecm` but can be hand-tuned. **Always edit via `pvecm`** or with extreme caution (use `corosync-cfgtool` to validate).

```ini
logging {
  debug: off
  to_syslog: yes
}

nodelist {
  node {
    name: pve-kubric-01
    nodeid: 1
    quorum_votes: 1
    ring0_addr: 10.0.200.21
  }

  node {
    name: pve-kubric-02
    nodeid: 2
    quorum_votes: 1
    ring0_addr: 10.0.200.22
  }

  node {
    name: pve-kubric-03
    nodeid: 3
    quorum_votes: 1
    ring0_addr: 10.0.200.23
  }
}

quorum {
  provider: corosync_votequorum
  expected_votes: 3
  # With 3 nodes, losing 1 node still has quorum (2/3)
  # Losing 2 nodes = no quorum = cluster stops
}

totem {
  cluster_name: kubric-cluster
  config_version: 3
  
  # Transport
  transport: udpu
  # udpu = unicast (works without multicast-capable switches)
  # knet = kronosnet (preferred for PVE 8.x, supports multiple links)

  # Crypto — encrypt and authenticate cluster traffic
  secauth: on
  crypto_cipher: aes256
  crypto_hash: sha256

  # Timing
  token: 5000
  # Token timeout in ms — node considered dead if no response in 5s
  # Default: 1000ms; increase for WAN or loaded networks
  
  token_retransmits_before_loss_const: 10
  # Retransmit attempts before declaring token loss
  
  join: 60
  # Join timeout in ms
  
  consensus: 6000
  # Consensus timeout — must be > token + 2*token_retransmit
  
  max_messages: 20
  # Max messages per token rotation
  
  send_join: 0
  # Delay before sending join — 0 for fast startup

  interface {
    ringnumber: 0
    bindnetaddr: 10.0.200.0
    # Bind to storage network for heartbeat
    # Using storage network avoids contention with management
    mcastport: 5405
  }
}
```

### 2.2 Dual-Link Configuration (Redundant Heartbeat)

For production, add a second heartbeat link over the management network:

```bash
# Add link1 on each node via pvecm
pvecm updatecerts --force

# Edit corosync.conf to add second link:
# nodelist → each node gets ring1_addr:
#   pve-kubric-01: ring1_addr: 10.0.100.21
#   pve-kubric-02: ring1_addr: 10.0.100.22
#   pve-kubric-03: ring1_addr: 10.0.100.23

# With knet transport (PVE 8.x), add:
# totem {
#   transport: knet
#   interface {
#     linknumber: 0
#     # Storage network
#   }
#   interface {
#     linknumber: 1
#     # Management network (backup)
#   }
# }
```

## 3. Quorum Configuration

### 3.1 Three-Node Quorum Rules

| Nodes Online | Votes | Quorum? | Action |
|-------------|-------|---------|--------|
| 3 | 3/3 | Yes | Full cluster operations |
| 2 | 2/3 | Yes | Cluster operational, HA failover works |
| 1 | 1/3 | No | Cluster stops, VMs freeze (by design) |

### 3.2 Expected Votes

```bash
# Check quorum status
corosync-quorumtool
# Expected:
#   Votequorum information
#   Expected votes:   3
#   Highest expected:  3
#   Total votes:       3
#   Quorum:            2
#   Activity blocked:  No
```

### 3.3 Quorum Adjustments (Emergency)

```bash
# EMERGENCY ONLY: If 2 nodes are permanently dead, allow single-node operation
# This is DANGEROUS — split-brain risk
pvecm expected 1

# Reset after repair
pvecm expected 3
```

## 4. pvecm Cluster Management Commands

### 4.1 Status & Health

```bash
# Cluster status
pvecm status

# List nodes
pvecm nodes

# Corosync ring status
corosync-cfgtool -s
# Expected: ring 0 active, no faults

# Check corosync process
systemctl status corosync
journalctl -u corosync --since "1 hour ago" --no-pager
```

### 4.2 Node Operations

```bash
# Remove a node (for replacement)
# 1. Shutdown the node to remove
# 2. From a remaining node:
pvecm delnode pve-kubric-03

# Re-add after repair
# From the re-added node:
pvecm add 10.0.100.21 --link0 10.0.200.23

# Force update certificates
pvecm updatecerts --force
```

### 4.3 Corosync Key Rotation

```bash
# Regenerate authkey (requires brief cluster restart)
# On node 1:
corosync-keygen -l
# Copies to /etc/corosync/authkey

# Sync to other nodes
scp /etc/corosync/authkey root@10.0.100.22:/etc/corosync/authkey
scp /etc/corosync/authkey root@10.0.100.23:/etc/corosync/authkey

# Restart corosync on ALL nodes simultaneously
clush -g all systemctl restart corosync
# Or serially:
ssh root@10.0.100.21 systemctl restart corosync
ssh root@10.0.100.22 systemctl restart corosync
ssh root@10.0.100.23 systemctl restart corosync
```

## 5. Fencing Configuration (STONITH)

### 5.1 Why Fencing Matters

Without fencing, a "split-brain" node might continue running VMs that the cluster has already restarted elsewhere, causing data corruption. STONITH (Shoot The Other Node In The Head) uses iDRAC IPMI to forcibly power off a non-responsive node.

### 5.2 Install Fence Agents

```bash
apt-get install -y fence-agents pve-ha-manager
```

### 5.3 Configure IPMI Fencing via iDRAC

```bash
# Test IPMI connectivity to each node
ipmitool -I lanplus -H 10.0.100.11 -U kubric-admin -P "${IDRAC_KUBRIC_PASS}" chassis status
ipmitool -I lanplus -H 10.0.100.12 -U kubric-admin -P "${IDRAC_KUBRIC_PASS}" chassis status
ipmitool -I lanplus -H 10.0.100.13 -U kubric-admin -P "${IDRAC_KUBRIC_PASS}" chassis status
```

### 5.4 Proxmox HA Fencing Configuration

File: `/etc/pve/ha/fence.cfg`

```ini
# IPMI fencing for each node
device:fence_ipmilan pve-kubric-01 \
  ipaddr=10.0.100.11 \
  login=kubric-admin \
  passwd="${IDRAC_KUBRIC_PASS}" \
  lanplus=1 \
  method=onoff \
  power_wait=10 \
  action=off

device:fence_ipmilan pve-kubric-02 \
  ipaddr=10.0.100.12 \
  login=kubric-admin \
  passwd="${IDRAC_KUBRIC_PASS}" \
  lanplus=1 \
  method=onoff \
  power_wait=10 \
  action=off

device:fence_ipmilan pve-kubric-03 \
  ipaddr=10.0.100.13 \
  login=kubric-admin \
  passwd="${IDRAC_KUBRIC_PASS}" \
  lanplus=1 \
  method=onoff \
  power_wait=10 \
  action=off
```

### 5.5 Test Fencing


## 6. Monitoring Corosync

### 6.1 Prometheus Metrics

```bash
# Install corosync exporter (or use node-exporter textfile collector)
cat > /usr/local/bin/corosync-metrics.sh <<'SCRIPT'
#!/bin/bash
OUTPUT="/var/lib/node_exporter/corosync.prom"
QUORATE=$(corosync-quorumtool -s 2>/dev/null | grep "Quorate:" | awk '{print ($2 == "Yes") ? 1 : 0}')
VOTES=$(corosync-quorumtool -s 2>/dev/null | grep "Total votes:" | awk '{print $3}')
EXPECTED=$(corosync-quorumtool -s 2>/dev/null | grep "Expected votes:" | awk '{print $3}')

cat > $OUTPUT <<EOF
# HELP corosync_quorate Whether the cluster has quorum
# TYPE corosync_quorate gauge
corosync_quorate $QUORATE

# HELP corosync_total_votes Current total votes
# TYPE corosync_total_votes gauge
corosync_total_votes $VOTES

# HELP corosync_expected_votes Expected votes for quorum
# TYPE corosync_expected_votes gauge
corosync_expected_votes $EXPECTED
EOF
SCRIPT

chmod +x /usr/local/bin/corosync-metrics.sh

# Run every 15 seconds
cat > /etc/systemd/system/corosync-metrics.timer <<'EOF'
[Unit]
Description=Corosync metrics collector

[Timer]
OnBootSec=30s
OnUnitActiveSec=15s

[Install]
WantedBy=timers.target
EOF

cat > /etc/systemd/system/corosync-metrics.service <<'EOF'
[Unit]
Description=Corosync metrics collector

[Service]
Type=oneshot
ExecStart=/usr/local/bin/corosync-metrics.sh
EOF

systemctl daemon-reload
systemctl enable --now corosync-metrics.timer
```

## 7. Troubleshooting

### 7.1 Common Issues

```bash
# Node can't join cluster
# Check: SSH connectivity, time sync, firewall
ssh root@10.0.100.21 date
chronyc tracking
iptables -L -n | grep 5405

# Corosync ports required:
#   UDP 5404-5405 (corosync)
#   TCP 22 (SSH for pvecm)

# Ring errors
corosync-cfgtool -s
# If ring shows "FAULTY":
#   - Check network cable
#   - Check IP address binding
#   - Verify MTU matches across all nodes

# Split brain recovery
# 1. Identify which partition has quorum
pvecm status
# 2. If no quorum, choose authoritative node
# 3. On non-authoritative nodes:
systemctl stop pve-cluster corosync
pmxcfs -l   # start in local mode
rm /etc/corosync/corosync.conf
rm /etc/pve/corosync.conf
kill $(pidof pmxcfs)
systemctl start pve-cluster
pvecm add 10.0.100.21 --link0 10.0.200.2X
```

### 7.2 Verify Everything

```bash
# Full health check
pvecm status
corosync-cfgtool -s
corosync-quorumtool
pvecm nodes
systemctl status corosync pve-cluster pvedaemon

# Check HA manager
ha-manager status
```
